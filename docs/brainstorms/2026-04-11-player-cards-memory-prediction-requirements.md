# Player Cards Phase 2: Memory + Prediction Loop

## Summary

Two tightly interlocked pieces ship as one v1:

1. **Memory layer** — cards anchor on a **last-10-games sparkline** that any reader sees immediately, plus localStorage-backed reader state (open count, first-seen, last-seen) that accumulates quietly on every card opened.
2. **Prediction loop** — followed-player cards (those in the My Players binder) surface a daily contextual micro-prediction generated by the build using stat-trigger rules. Picks lock at first pitch of the player's next game. Yesterday's picks resolve silently each morning from final boxscores. A per-player scoreboard lives on the card back; an aggregate summary lives in the binder header.

The two halves form a loop: **open a card → see the trend and your touches → make (or revise) a prediction → return tomorrow → see the result**. v1 is not a v1 without the full loop.

## Problem Frame

Phase 1 shipped today: deeplinks, 30-team active-roster cards, Tier 1 click affordance, the dark binder My Players section. What's missing is a **reason to come back tomorrow** and a **reason to care about a specific player over time**.

Right now every card is a cold read. Opening the same card two days in a row surfaces the same front with no acknowledgement that you've been here before, no signal of what's changed, no stake in tomorrow. The product is a well-designed briefing but not yet a daily ritual that pulls.

Phase 2 turns the card from a lookup into a relationship. The sparkline gives every reader an immediate "what's this player been doing lately" moment on any card. The prediction loop gives followed players a tomorrow-hook. Reader state lets the card acknowledge the relationship.

## Requirements

### Memory layer

- **R1 [HIGH]** — Cards carry per-reader state in localStorage: `openCount`, `firstSeenDate`, `lastSeenDate`, keyed by pid. No backend.
- **R2 [HIGH]** — Every card surfaces a **last-10-games sparkline** computed from the MLB gamelog endpoint. Hitters: OPS per game. Pitchers: GameScore per game. The sparkline is the memory anchor — visible to any reader, no reader state required.
- **R7 [HIGH]** — Reader-state memory accumulates on **every** card opened, followed or not. Memory is a universal layer; the followed/not distinction only applies to the prediction layer.
- **R14 [MEDIUM]** — The card back surfaces the reader's touches in the Morning Lineup voice: "You've opened this 14 times · first seen Apr 4."

### Prediction loop

- **R3 [HIGH]** — The back of every followed-player card surfaces a single daily contextual micro-prediction tile with question text and a YES/NO tap.
- **R8 [HIGH]** — Prediction prompts appear **only** on followed-player cards. Random cards opened from team pages still accumulate memory but show no prediction tile.
- **R9 [HIGH]** — The daily build selects a contextual question per followed player using a **stat-trigger rule system** (slump-snap, hot-streak continuation, pitcher-matchup, milestone-watch, etc.). Fallback when no rule fires: a generic question appropriate to role ("Will he get a hit today?" / "Will he record 5+ Ks?").
- **R10 [HIGH]** — Each prediction record carries: `question_text`, `resolution_rule` (how to check next day's boxscore), `role_tag` (hitter/pitcher), `context_tag` (which stat-trigger fired).
- **R13 [HIGH]** — **DNP handling** — if the player does not play, the prediction is pushed. No record change, fresh question generated next morning. DNP does not roll over or auto-lose.
- **R17 [HIGH]** — **Pick lock timing** — picks can be revised freely until first pitch of the player's next scheduled game. After first pitch, the pick is locked. This requires the build to know the player's next game time.
- **R15 [HIGH]** — **Silent resolution** — every daily build runs a resolution pass: for every open prediction from the previous day, check the final boxscore, compute the result, update the scoreboard. Resolution happens whether or not the reader was present.
- **R16 [MEDIUM]** — Predictions that resolved while the reader was away are marked with a subtle "you missed this one" indicator on the scoreboard. Still counted toward the record.

### Scoreboard

- **R4 [HIGH]** — A personal scoreboard accumulates per-player prediction history in localStorage: total picks, correct picks, current streak, result of most recent pick.
- **R11 [HIGH]** — **Per-player scoreboard** lives on the back of each followed-player card: "You're 7-for-12 on Happ this season · currently 3-of-last-5."
- **R12 [HIGH]** — **Aggregate scoreboard summary** lives in the My Players binder header: overall season record across all followed players + current streak.

### Shipping constraint

- **R6 [HIGH]** — v1 ships the full loop (sparkline + reader state + prediction + scoreboard) as one unit. A partial ship (memory without prediction, or prediction without scoreboard) is not a v1 win.

### Stretch

- **R5 [MEDIUM]** — End-of-season "Your Season" recap built from accumulated localStorage state (most-opened player, best prediction streak, total picks made, record vs. MLB average, hot/cold takes you nailed). Spotify-Wrapped energy. Can ship after v1 lands if timeline pressures demand.

## Success Criteria

- **SC1 [HIGH]** — Opening any card immediately surfaces a last-10-games sparkline, regardless of whether the reader has seen the player before.
- **SC2 [HIGH]** — Opening a followed-player card the morning after making a prediction shows the resolved result on first view, with no extra navigation.
- **SC3 [HIGH]** — After two weeks of daily use, the per-player scoreboard on a followed player's card shows a meaningful record (W-L-Push and recent streak).
- **SC5 [HIGH]** — The full loop — open → sparkline → predict → return next morning → see result — runs with zero friction and zero modal navigation.
- **SC6 [HIGH]** — Following a player in `/settings/` adds them to the prediction loop the next morning's build. Unfollowing removes them cleanly.
- **SC8 [HIGH]** — Opening the My Players binder surfaces aggregate record + current streak in the header **before** any card is opened.
- **SC10 [HIGH]** — A reader returning after multiple days away sees all intervening predictions already resolved in the scoreboard. Missed-while-away predictions are marked honestly but still counted.
- **SC7 [MEDIUM]** — Daily questions feel **written**, not templated. A reader scanning a week of questions should not feel like they're all the same Mad Libs.
- **SC9 [MEDIUM]** — Any returning reader can see at a glance how often they've checked a given player this season.
- **SC4 [MEDIUM]** — The end-of-season recap is interesting enough that readers share it.

## Scope Boundaries

### Out of scope for v1

- **[HIGH]** Cross-device sync via auth. v1 is localStorage-only. A user switching devices starts fresh. Auth and sync are a post-v1 concern (and tie into existing Supabase infrastructure already used by followed_players and followed_teams — that integration is a later decision).
- **[HIGH]** Server-side prediction storage. All prediction state lives in localStorage. No database writes for picks.
- **[HIGH]** Prediction prompts on non-followed-player cards. Following is the gate. Random cards get memory only, no prediction tile.
- **[HIGH]** Per-day stat snapshot tape. The last-10-games sparkline is computed from the live MLB gamelog endpoint — no persisted snapshot artifact is needed, which removes a significant storage and janitor concern.
- **[MEDIUM]** Memory or prediction for non-active-roster players (IL, 40-man non-active, prospects). v1 covers the 779 active players already in the pipeline. IL/prospects stay deferred as in Phase 1.
- **[MEDIUM]** Reader-authored custom predictions ("let me write my own question"). The editorial voice chooses the question — this is a feature, not a limitation.

### Deferred to planning

- **[MEDIUM]** The **stat-trigger rule starter set** — which contextual question templates ship at v1 (slump-snap, hot-streak, pitcher-matchup, milestone-watch, generic-role-fallback, etc.) is a content-library decision that planning can scope against build-time complexity. The brainstorm commits to the *mechanism*, not the starter list.
- **[LOW]** Exact sparkline dimensions, colors, and responsive behavior on mobile.
- **[LOW]** Exact scoreboard layout on card back and binder header (prose vs. numeric table).
- **[LOW]** Whether the season recap is a dedicated URL (`/recap/`) or an overlay on the binder at season's end.

## Key Decisions & Rationale

| Decision | Rationale |
|---|---|
| **Memory + prediction ship together** | The loop doesn't work as halves. Memory alone is "nice to know what changed"; prediction alone has nothing to anchor. Together they create a daily ritual. |
| **Followed-only predictions, universal memory** | Predictions require user investment (who do I care enough to pick on?). Memory is free and universal — every card gets more useful the more you use it. The split respects user attention without restricting the product. |
| **Editorial smart-pick for the question** | Aligns with Morning Lineup's editorial voice (ideation #3). Turns the card from a stat aggregator into a paper with a take. Deterministic (no LLM at runtime), cheap per render. |
| **Last-10 sparkline, not "delta since last visit"** | Simpler, more universal, and reuses existing gamelog infrastructure. Every reader gets the same signal regardless of history. Removes the need for a persisted snapshot tape entirely. |
| **localStorage only for v1** | Zero backend investment. Ships faster. Auth/sync is a real post-v1 feature with its own brainstorm. The tradeoff (cross-device) is acceptable because Morning Lineup is primarily a laptop-with-coffee ritual. |
| **Silent resolution with visible "missed" marks** | The scoreboard stays accurate regardless of attendance, so records mean something. Honest absence indicators avoid the "feels like cheating" problem. |
| **Picks lock at first pitch** | Matches fan intuition (you can change your mind until the game starts). Requires schedule lookup but the build already fetches schedule data for other sections. Gives the loop an honest stakes. |
| **DNP = push** | Rest days and postponements happen too often for any other rule to feel fair. Fresh question tomorrow keeps the loop moving. |

## Open Questions for Planning

- Which stat-trigger rules ship at v1? Suggested starter set: slump-snap (BA <.200 over last 10 for hitters / ERA >5 over last 3 starts for pitchers), hot-streak-continuation (OPS >1.000 over last 10 / 10+ K in last start), milestone-watch (approaching 10/20/30 HR, 100 K, etc.), pitcher-day (SP starts today), generic-role-fallback.
- How is "next scheduled game time" fetched — reuse existing schedule fetch in `build.py` or add a dedicated call?
- Does the daily build pipeline gain a new step (resolution pass), or does it fold into the existing per-team build? The resolution pass is cross-cutting — resolves predictions for players across many teams.
- What happens if the reader follows a player mid-season with no prior open history — the sparkline still shows (universal) but reader-state fields are empty. UX for first-open state.
- How does the stat-trigger rule selector handle a player who triggers multiple rules on the same day (e.g., both slump-snap and pitcher-matchup)? Priority order, random, or editorial hand-pick?
- Storage shape for localStorage: one flat `morning-lineup.cards` object, or namespaced keys?
- Where does the binder header summary live relative to the existing "Approved" stamp and followed count — inline or below the rule?
