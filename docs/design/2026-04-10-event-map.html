The Event Map

VOL. I · 2026-04-10 · LIVING DOCUMENT

Every "how often should this refresh?" question is really "what event should trigger this?" in disguise. If something upstream already observes the state change, the downstream doesn't need its own timer — it just needs to be kicked. Map the events first, then wire the consumers.

VThe Gap List

Ranked by reader-value-per-effort

01 Lineup posting → scouting report refresh High

Event: boxscore.battingOrder becomes non-empty on today's game feed (~1 hour before first pitch).

Today: scouting report is rendered at 5am morning build from probable-pitcher data only. No starting lineups yet. Frozen all day.

Could do: subscribe to the Cubs game feed (already running post-ship for the widget) and when a lineup appears, refresh the scouting report's "opposition lineup" card. One new derived subscriber, kicked by the Cubs widget — same timerless pattern as standings.

Why it matters: lineups are the single most reader-interesting pregame information. Getting them live changes the briefing from "morning preview" to "game-day companion."

02 Date rollover → slate reset Medium

Event: clock ticks past midnight local time while a tab is open.

Today: tab opened at 11:55pm shows yesterday's slate forever. User has to refresh.

Could do: tiny time-watcher that fires once at midnight and calls slateHandle.refreshNow(). The slate's endpoint function already uses today() so the next fetch hits the new date automatically. Plus: standings kicks once, late-finals unsubscribes and a new one subscribes for the new "yesterday."

Why it matters: low-probability scenario but the fix is ~10 lines and removes a hidden footgun.

03 Transaction feed → pressbox live Medium

Event: /v1/transactions?date={today} gets new entries during the day.

Today: pressbox is frozen at morning build. A 2pm trade announcement doesn't show until tomorrow.

Could do: low-frequency subscriber (every 10 min) polling today's transactions and patching the pressbox table. Farm section becomes a derived listener for call-ups/send-downs.

Why it matters: transactions are bursty; you don't want to miss the Cubs making a move. Absolute win over "reload tomorrow morning" is smaller than #1 but still real.

04 Injury list → pressbox + form guide Medium

Event: /v1/teams/{id}/roster?rosterType=injuryList returns new entries.

Today: morning build only.

Could do: same low-frequency subscriber as #3, piggybacking on the transactions poll cadence. Pressbox IL side updates. If a hot form-guide hitter lands on the IL, flag him.

Why it matters: injuries propagate slower than transactions but the freeze is most user-hostile here — you find out your guy is hurt when you open the app tomorrow.

05 Probable pitcher change → scouting + slate Low

Event: /v1/schedule?hydrate=probablePitcher diff from what morning build captured.

Today: nothing watches. A 10am pitcher swap means the 5am scouting report is wrong all day.

Could do: piggyback on the existing slate subscriber — it already hydrates probablePitcher. Diff check in the slate's onData; if any gamePk's probable pitcher changed, kick a scouting-report refresh subscriber.

Why it matters: rare event. But when it happens, the current state is embarrassingly wrong — so leverage per incident is high even if frequency is low.

06 Weather update → pregame panel Low

Event: venue weather changes between morning build and first pitch.

Today: morning build only. Rain at 5pm with 7pm first pitch still shows "clear 72°."

Could do: 30-min pre-game subscriber that refreshes weather from MLB's venue endpoint.

Why it matters: rarely decisive; most readers will see actual conditions by first pitch. Lowest leverage of the gaps.

VIThree Patterns the Map Surfaces

The slate is an event fountain

It already polls once, sees every today-game transition, and can kick an arbitrary number of derived subscribers. Standings is the first consumer. Scouting-report-on-lineup and rival-results on transitions should both reuse the same pattern. The slate's onData is becoming the central event dispatcher for today-game events.

The morning build is an event too

Just a one-shot, full-page variant. Anything it populates is, in principle, a candidate for live refresh by finding the upstream source event. The question "should this be live?" becomes "is the upstream event worth watching at client-side cadence?"

Some sections have no live source at all

History, This Day In, Editorial Lede. These are correctly frozen. The map helps distinguish "frozen because there's no event" from "frozen because nobody wired the event yet." Don't waste effort trying to make the first category live.

VIINext Moves

  • Ship unified-live-core first (docs/plans/2026-04-10-005-feat-unified-live-core-plan.md).
  • After ship, pick gap #1 (lineup → scouting) and #2 (date rollover). Both cheap; #1 is high reader value, #2 closes a real footgun.
  • Gaps #3–6 are backlog material. Re-read this map when planning the next editorial pass and see which ones have climbed in value.
